# CodeGraph 项目立项书

**项目代号**：CodeGraph
**文档版本**：v1.1
**日期**：2026-07-07

---

## 一、背景

团队当前全面推进 AI Coding，实践已验证：为核心模块提供结构化上下文文档（`.codex/skills/`），能显著提升 AI 工具在具体工程任务中的准确性。当前 skill 文件由 AI 按统一规范生成，质量达到可用水准。

随着覆盖模块增多，三个结构性问题出现：

1. **平铺堆积**：所有 skill 堆在同一目录层级，无领域分层，数量增长后导航失效
2. **文档孤岛**：每个 skill 独立存在，AI 每次任务只拿到一个碎片，无法感知跨模块依赖链
3. **关系隐式**：skill 正文中虽然提到了其他模块，但这些依赖关系没有被结构化记录，AI 必须自行猜测

本立项书在确定技术方案前，先完整评估了三条主流路线的适用性。

---

## 二、方案比较

### 方案 A：向量数据库 RAG

**原理**：将所有 skill 文件切成 chunks，嵌入向量，提问时做相似度检索，把最相关的若干 chunks 拼进上下文。

**优势**：
- 对大量非结构化文本检索效果好
- 无需人工维护文档关系
- 工程实现成熟，开箱即用

**局限**：

第一，**靠统计距离猜测相关性**。向量相似度衡量的是文字表面的语义接近程度，而不是代码依赖关系。`formulacommands.ts` 和 `formulas.ts` 在代码逻辑上强依赖，但它们的文字描述用词可能差异较大，向量距离不一定近，导致检索时漏掉关键上下文。

第二，**每次查询重新推导，没有积累**。Karpathy 在 llm-wiki 中明确指出这个问题：RAG 是"每次从原始文档重新发现知识"，没有编译，没有沉淀。团队每次让 AI 分析公式系统，AI 都要重新拼凑碎片，不存在"上次分析结果"的积累。

第三，**关系仍然是隐式的**。切块之后，`UpstreamFormulaCommand` 依赖 `formulas.hasFormulaPrefix` 这个事实存在于文字中，但不是结构化数据。AI 看到的是"这两段话语义相近"，而不是"A 调用了 B"。

第四，**需要额外基础设施**。向量数据库（Pinecone、Weaviate、pgvector 等）的接入、维护、更新策略，都需要额外工程投入，且与团队现有的纯文件工作流不兼容。

**结论**：向量 RAG 适合大量自由格式文本的检索场景（如 Slack 消息、会议记录），不适合有清晰依赖结构的代码模块文档。

---

### 方案 B：LLM Wiki 模式（Karpathy 提案）

Karpathy 在 [llm-wiki](https://github.com/karpathy/llm-wiki) 中描述的模式与本项目诉求高度吻合，值得单独分析。

**核心思想**：LLM 不只是检索原始文档，而是持续维护一个结构化的 Markdown wiki——每加入一个新源文档，LLM 读取、提炼、更新相关页面、建立交叉引用、标记矛盾。Wiki 是持续积累的编译产物，而不是每次查询时从零推导。

这个思路解决了 RAG 的积累问题：**知识被编译一次，持续维护，不重复推导**。

> "Ask a subtle question that requires synthesizing five documents, and the LLM has to find and piece together the relevant fragments every time. Nothing is built up."
> — Karpathy, llm-wiki

这正是 RAG 的问题，也正是本项目要解决的问题。

**LLM Wiki 对本项目的启示**：
- Wiki 的维护成本应该由 LLM 承担，人只负责策源和质量把关
- 良好的 schema（对应 OKF 的文档格式规范）是 LLM 成为"有纪律的 wiki 维护者"的关键
- 文档间的交叉引用和 graph view 是知识库有效性的核心指标
- 文档存 Git，变更历史、协作、Code Review 全部复用现有工具链
- 人负责策源和质量方向，LLM 承担所有维护性的书写工作

**LLM Wiki 的局限**：

llm-wiki 是一个意图描述文档，没有具体实现。它提到的三层架构（raw sources / wiki / schema）和具体操作流程（ingest / query / lint）是正确的方向，但在代码工程场景下需要具体化：

- 原始来源不是文章、论文，而是代码文件和现有 skill 文档
- Schema 不只是格式规范，还需要包含代码模块文档的特定结构（快速定位表、数据流、排查路径等）
- 交叉引用的发现不只靠语义，还需要结合代码的实际调用关系

**结论**：LLM Wiki 提供了正确的方向，需要针对代码工程场景具体实例化。OKF 方案正是这个实例化的结果。

### 方案 C：OKF 结构化知识图谱（本项目选择）

**原理**：LLM 阅读代码文件和现有 skill，按 OKF 格式规范生成结构化文档，在文档中显式声明模块间的依赖关系（`Related Concepts` 章节），构建可图遍历的知识网络。

**OKF 与 LLM Wiki 的继承关系**：

OKF 可以理解为 LLM Wiki 在代码工程场景下的具体实现，继承了其核心思想，并针对代码文档的特点做了具体化：

| LLM Wiki 的原则 | OKF 的对应实现 |
|---|---|
| Wiki 是持续积累的编译产物 | Pass 1 生成文档，Pass 2 增量增强，增量保护确保不退化 |
| Schema 约束 LLM 的输出格式 | OKF frontmatter 规范 + 系统提示词（`jsInstruction.md`）|
| 交叉引用是知识库的核心价值 | `Related Concepts` 章节，硬链接，驱动图谱的边 |
| Graph view 展示知识形状 | `viz.html`，Cytoscape.js，节点大小反映枢纽程度 |
| Git 是最好的版本管理 | 文档存文件系统，天然 Git 追踪每次变更 |
| 人负责策源，LLM 做维护 | 人确认 `domain_manifest.json`，LLM 完成所有文档写作 |

**OKF 相对于 RAG 的核心技术优势**：

**优势 1：关系是显式声明的，不是猜测的**

```
RAG 的检索逻辑：
  "哪个 chunk 和这个问题最相似？" → 向量距离（统计猜测）

OKF 的检索逻辑：
  "从这个模块出发，Related Concepts 链接到哪里？" → 图遍历（精确导航）
```

`formula_command.md` 里写明了：依赖 `formula_utils`，依赖 `command_router`，关联 `falcon/calc/evaluator`。这不是 AI 在查询时猜出来的，而是在文档生成时由 LLM 阅读代码后显式声明的。这条信息是稳定的、可审计的、可维护的。

**优势 2：上下文精确收集，不靠碰运气**

AI 处理"公式输入后结果不更新"的排查任务时：

```
RAG 路径：
  向量检索 → 找到最相似的 3 个 chunk
  → 可能漏掉 evaluator 部分 → 答案不完整

OKF 路径：
  formula_command → Related Concepts
  → 收集 command_router + formula_utils + evaluator
  → 完整的四层链路进入上下文 → 答案准确
```

**优势 3：知识是积累的，不是每次重新推导**

和 LLM Wiki 一样，OKF 文档一旦生成就是持续积累的资产。下次 AI 回答关于公式系统的问题，不需要重新分析代码，直接读已经编译好的结构化文档。

**优势 4：对人可读，可 Code Review，可手动修正**

向量数据库里的向量不可读，无法 Review，无法手动修正。OKF 文档是普通 Markdown 文件，工程师可以直接阅读、修改、提 PR。生成质量有问题的部分可以人工校正，校正的历史在 Git 里留存。

**优势 5：零额外基础设施依赖**

没有向量数据库，没有嵌入模型，没有向量更新管道。只需要一个 LLM API Key 和文件系统。团队的现有工具链（Git、CI、编辑器）可以直接使用。

### 三方案对比总结

| 维度 | 向量 RAG | LLM Wiki | OKF（本项目）|
|---|---|---|---|
| 知识是否积累 | ❌ 每次重新推导 | ✅ 持续编译积累 | ✅ 持续编译积累 |
| 关系是否显式 | ❌ 向量距离隐式 | ⚠️ 依赖 schema 设计 | ✅ `Related Concepts` 硬链接 |
| 检索准确性 | ⚠️ 语义相似 ≠ 代码依赖 | ✅ 基于结构 | ✅ 图遍历精确收集 |
| 对人可读可审计 | ❌ 向量不可读 | ✅ Markdown | ✅ Markdown + YAML |
| 基础设施需求 | ❌ 需要向量数据库 | ✅ 纯文件 | ✅ 纯文件 |
| 代码工程适配度 | ⚠️ 通用，非专门设计 | ⚠️ 通用，需自行实例化 | ✅ 专门针对代码模块文档 |
| 与现有 skill 兼容 | ⚠️ 需切块、嵌入 | ⚠️ 需重新设计 schema | ✅ 直接继承现有格式 |

**选择 OKF 的理由**：

OKF 是 LLM Wiki 思路在代码工程场景下的具体实现，解决了 RAG 的积累缺失和关系隐式两个根本问题，同时：团队已有的 skill 规范可以直接继承，不需要从零重新设计文档格式；零额外基础设施依赖，接入成本最低；文档对人可读可审计，与团队现有工程文化兼容。

## 三、项目目标

在不改变现有 skill 生成流程的前提下，为 skill 文件体系补上三项缺失能力：

| 能力 | 解决的问题 | 预期效果 |
|---|---|---|
| 领域分层组织 | 平铺堆积 | 数量增长后仍可快速导航 |
| 显式依赖关系 | 文档孤岛 | AI 任务上下文完整率提升 |
| 图谱可视化 | 关系隐式 | 架构依赖一目了然，枢纽模块可识别 |

---

## 四、预期收益

### AI Coding 任务准确性提升

当前跨模块任务（如"修改公式系统不影响单元格渲染"）中，AI 因上下文不完整给出错误方向的比例约 30~40%（主观估计，基于团队日常经验）。OKF 图遍历机制确保相关模块文档全部进入上下文，预期此类错误率降低 50% 以上。

### 排查效率提升

当前工程师排查跨模块 bug 时，需要自行梳理调用链路（平均 30~60 分钟）。OKF 的数据流章节和 Related Concepts 链接提前将链路记录，AI 辅助排查时可直接利用，预期排查时间缩短 40~60%。

### 新成员上手加速

OKF 知识图谱是对工程架构认知的可视化沉淀。新成员可以通过 `viz.html` 的图谱视图，快速理解"哪些模块是核心（枢纽节点）、各子系统之间如何交互"，这是当前文档体系无法提供的全局视角。

### 文档维护成本趋近于零

与 LLM Wiki 的理念一致：LLM 不会忘记更新交叉引用，不会因为维护负担增长而放弃。CI 触发的增量更新确保代码变更后文档自动同步，人工介入只用于质量审核。

## 五、实施计划

### Phase 1（验证期，2 周）

以 1 个核心子系统为试点，跑通完整 Pipeline：

- Pass 0：LLM 分析领域归属，生成 `domain_manifest.json`
- 人工确认领域结构（唯一的人工介入节点）
- Pass 1：文档迁移重组，规范化为 OKF 格式
- Pass 2：LLM 推断并写入模块间 `Related Concepts` 关系
- 后处理：生成层级索引 + `viz.html` 图谱

**成功标准**：图谱覆盖试点子系统 80% 以上模块，推断关系准确率 > 70%

### Phase 2（推广期，2 周）

- 全量模块覆盖
- CI 集成：PR 合并后自动增量更新受影响模块的文档
- skill 导出：OKF 文档自动转换为 `.codex/skills/` 格式，AI Coding 工作流无感知

### Phase 3（持续演进）

- 图遍历查询 API：给定任务描述，自动返回相关文档集合
- 知识图谱健康度监控：检测文档与代码的一致性
- 影响分析：代码变更时，自动识别哪些图谱文档需要更新

## 六、技术选型

| 决策 | 选择 | 理由 |
|---|---|---|
| 文档格式 | OKF（YAML frontmatter + Markdown）| 机器可读 + 人可读 + Git 追踪变更 |
| LLM 框架 | 原生 OpenAI 兼容 API | 不绑定特定框架，支持 DeepSeek / 通义 / OpenAI |
| 语言 | TypeScript | 团队主要技术栈，可接入现有工具链 |
| 关系表达 | 显式硬链接（非向量）| 精确、可追溯、支持图遍历、不依赖向量数据库 |
| 文档存储 | 文件系统（非数据库）| Git 可追踪每次变更，可 Code Review |

## 八、总结

现有 skill 文件是一批高质量的孤立文档。CodeGraph 要做的，是在 LLM Wiki 思路的指导下，用 OKF 格式将这些文档组织成一张网——有层级、有显式依赖关系、可被 AI 精确遍历。

选择 OKF 而非向量 RAG 的根本原因：**代码工程的知识结构是图，不是向量空间。依赖关系是精确的、有方向的、可声明的，不是靠统计猜测的。** 用图结构表达图问题，比用向量距离近似图关系，在准确性、可维护性、可追溯性上全面占优。

OKF 不是一个全新的发明，而是对 LLM Wiki 模式的一次工程化落地——把一个正确的思路，实现为一套团队可以直接使用的工具链。
